## 问题

Markdown 分块策略：chunk size 500、overlap 50 是如何选的？

## 参考答案

系统中 Markdown 文档的分块参数为 `chunk_size=500`、`chunk_overlap=50`，这组值在 `chunker.py:45-48` 的 `chunk_markdown` 函数中定义，并在 `bm25_index.py:240` 的 `build_from_knowledge_dir` 中被调用。选择这组参数是多方面权衡的结果。

**chunk_size=500 字符：平衡上下文丰富度与嵌入精度。** 500 个中文字符约等于 250 个 token（中文通常 1 字 ≈ 0.5 token）。系统使用的嵌入模型为 BAAI/bge-small-zh-v1.5，其最大输入长度为 512 token。500 字符的 chunk 刚好能完整送入嵌入模型而不会被截断，同时保留了足够的上下文信息供向量检索使用。如果 chunk_size 过大（>1000 字符），嵌入向量会因为信息过于稀释而降低检索精度；如果过小（<200 字符），则丢失上下文语义，导致向量表示不准确。

**chunk_overlap=50 字符：防止边界信息丢失。** 10% 的重叠比例（50/500）是分块策略中的常见启发式值。重叠确保了被分割点切断的句子或概念能在相邻两个 chunk 中都出现，避免检索时遗漏恰好落在边界上的关键信息。例如一段关于"OAuth 认证流程"的描述如果被 chunk 边界截断，50 字符的重叠可以让完整的流程描述至少出现在一个 chunk 中。

**分隔符层级：保持语义完整性。** `chunk_markdown` 使用 LangChain 的 `RecursiveCharacterTextSplitter`，分隔符优先级为（`chunker.py:61`）：`\n\n`（段落分隔）> `\n`（换行）> `。`（句号）> `！？；`（中文标点）> `. ! ? ;`（英文标点）> `, ，`（逗号）> ` `（空格）> `""`（字符级）。这种层级设计确保分块优先在段落边界切割，其次在句子边界切割，最大程度保持每个 chunk 内的语义连贯性。`keep_separator=True` 参数保证分隔符不会被丢弃。

**向量检索适配。** 分块后的文本同时用于 BM25 关键词索引（`bm25_index.py:240`）和 Milvus 向量检索（`vector_store.py:97` 使用 COSINE 相似度）。500 字符的 chunk 在两种检索方式下都能获得较好的效果：对 BM25 来说足够包含关键词上下文，对向量检索来说不会因文本过长而稀释语义信号。

需要指出的是，这组参数是经验性的起始值，而非经过系统化调优的结果。生产环境通常需要通过 A/B 测试来确定最优的 chunk_size 和 chunk_overlap 组合。

## 回答要点

- chunk_size=500 是嵌入模型限制（512 token）和上下文丰富度之间的平衡点
- chunk_overlap=50（10% 重叠）是防止边界信息丢失的常见启发式值
- 分隔符层级设计确保优先在语义边界（段落 > 句子 > 子句）处切割
- 500 字符同时适配 BM25 关键词检索和 Milvus 向量检索两种模式
- 这组参数是经验值，生产环境需要 A/B 测试优化

## 常见追问

**Q: 为什么不用更大的 chunk_size，比如 1000？**

A: 主要受限于嵌入模型。bge-small-zh-v1.5 的最大输入为 512 token（约 1000 中文字符），超过会截断。虽然理论上可以用更大的 chunk_size 并在嵌入时截断，但这会导致 chunk 尾部信息丢失，嵌入向量无法完整表示 chunk 内容。500 字符留出了约 50% 的余量，确保即使分隔符导致实际 chunk 略超 500 字符也不会被截断。

**Q: overlap 设为 0 会有什么问题？**

A: overlap=0 意味着相邻 chunk 之间没有重叠。如果一段关键信息（比如一个完整的 API 参数说明）恰好被分隔符切割到两个 chunk 的边界上，那么两个 chunk 都无法完整表示这段信息，检索时可能遗漏。50 字符的重叠提供了约 1-2 个句子的缓冲区，显著降低了这种边界效应。

**Q: 为什么不按语义分块（如按标题拆分）而非用固定字符数？**

A: 语义分块（如按 Markdown 标题拆分）是更优的方案，但实现复杂度高且 chunk 大小不可控——某些章节可能只有 50 字符，某些可能超过 5000 字符。当前的 `RecursiveCharacterTextSplitter` 方案在简单性和效果之间取得了较好的平衡，且分隔符层级已经优先在段落边界切割，部分实现了语义感知。
