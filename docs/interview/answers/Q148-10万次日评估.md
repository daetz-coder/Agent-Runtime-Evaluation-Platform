## 问题

如果日评估量从 100 次增长到 10 万次，架构上需要改什么？

## 参考答案

当前架构是面向低并发原型设计的，日评估量从 100 次跳到 10 万次意味着峰值 QPS 从不到 1 上升到约 1.2（均匀分布）或更高（突发场景），同时日均 LLM 调用量从约 600 次暴增到 60 万次。需要从五个层面进行改造：

### 1. 数据库：SQLite 必须替换为 PostgreSQL

当前默认数据库是 `sqlite+aiosqlite`（`app/core/config.py:37`），SQLite 的写锁机制在并发场景下会成为致命瓶颈——单写多读模型下，并发写事务会排队等待，10 万次评估意味着 10 万次写入评估结果，SQLite 无法承受。需要：

- 迁移到 PostgreSQL，使用 `asyncpg` 作为异步驱动，连接池大小设为 20-50
- 为 dashboard 和 report 查询添加只读副本（read replica），分担查询压力
- 评估结果表按时间分区（月分区），便于数据归档和查询加速
- 为 `evaluations.task_id` 和 `evaluations.status` 添加复合索引，加速轮询查询

### 2. 任务队列：BackgroundTasks 必须替换为 Celery

当前 API 层的 dispatch 逻辑是先尝试 Celery、失败则回退到 FastAPI 的 `BackgroundTasks`（`app/api/v1/endpoints/evaluation.py:186-202`）。`BackgroundTasks` 运行在 API 进程内，没有重试、没有持久化、没有水平扩展能力。10 万次日评估下：

- Celery 成为强制依赖，使用 Redis 作为 broker（`app/core/config.py:40` 已有 `REDIS_URL` 配置）
- Worker 数量根据队列深度自动伸缩（Kubernetes HPA 或 Celery Autoscaler）
- 配置死信队列（DLQ）处理反复失败的任务
- 每个评估任务设置超时（当前 `EVAL_BATCH_SIZE=10`，`config.py:97`），防止单任务阻塞 worker

### 3. LLM 调用成本与缓存策略

6 个评估器（planning、tactical、tool_use、memory、replan、retrieval）对每条 trajectory 各调用一次 LLM（`app/graphs/evaluation_graph.py:448-455`），10 万次日评估 = 60 万次 LLM 调用。当前的 Redis 缓存（`app/evaluators/base.py:294-372`）使用 `llm:{evaluator}:{model}:{hash}` 作为 key、24h TTL（`config.py:46`），在重复 trajectory 场景下能显著降低实际调用量。但需要额外优化：

- 批处理相似 prompt，合并多个评估请求共享同一 LLM 上下文
- 对非关键维度使用更便宜的模型（如 DeepSeek Chat 替代 GPT-4）
- Redis 内存管理：每天新增约 60 万条缓存 entry，24h TTL 后自动淘汰，内存增长可控（约 2-3 GB）
- 引入缓存命中率监控指标，目标是 30%+ 的缓存命中率

### 4. 水平扩展架构

当前单进程 asyncio 模型无法扩展。改造为：

- **无状态 API 层**：多个 FastAPI 实例放在负载均衡器后面，共享 Redis 和 PostgreSQL
- **分布式去重**：当前 `try_claim_stream`（`app/services/evaluation_service.py:69-81`）已经实现了 Redis-based 去重 + 本地 fallback 的双层模式，扩展到多实例后 Redis 去重成为唯一可靠路径
- **Celery Worker 独立扩展**：worker 数量与 API 实例数解耦，根据队列深度独立伸缩
- **共享状态**：所有会话状态、stream claim、缓存都通过 Redis 共享，避免 sticky session

### 5. 监控与告警

当前已有 Prometheus 指标（`app/core/metrics.py`），包括 `EVALUATION_COUNT`（Counter）、`EVALUATION_DURATION`（Histogram）、`EVALUATION_SCORE`（Histogram）、`LLM_CALL_COUNT`、`LLM_CALL_DURATION` 等。10 万量级下需要补充：

- Celery 队列深度告警（阈值 > 1000）
- LLM 调用 P99 延迟告警（阈值 > 10s）
- 评估失败率告警（阈值 > 5%）
- Redis 内存使用率告警（阈值 > 80%）
- PostgreSQL 连接池使用率告警

总结：核心改造路径是 SQLite → PostgreSQL、BackgroundTasks → Celery、单进程 → 多实例无状态架构，辅以缓存优化和完善的监控告警体系。

## 回答要点

- SQLite 写锁是并发写入的硬瓶颈，必须迁移到 PostgreSQL + asyncpg
- BackgroundTasks 无重试、无持久化，Celery 成为强制依赖
- 6 评估器 x 10 万次 = 60 万次 LLM 调用/天，缓存策略和模型选择是成本控制关键
- 无状态 API + Redis 共享状态 + Celery 独立扩展是水平扩展三要素
- `try_claim_stream` 的本地锁 fallback 在多实例下失效，必须统一走 Redis
- Prometheus 指标已有基础，需要补充队列深度和 LLM 延迟告警

## 常见追问

**Q: 为什么不直接用 Kafka 替代 Celery + Redis？**

A: Kafka 适合事件驱动的大数据流场景，但评估任务的特点是低吞吐（10 万/天 ≈ 1.2 QPS）、需要任务级重试和结果追踪。Celery + Redis 在这个量级下更轻量、运维成本更低，且项目已有 Redis 基础设施。只有当日评估量突破百万级、或需要事件溯源时才考虑 Kafka。

**Q: 如何处理评估任务的幂等性？**

A: `try_claim_stream`（`evaluation_service.py:69`）已经实现了基于 Redis `SET NX` 的幂等保证——同一个 `evaluation_id` 不会被重复执行。在多实例场景下，这个 Redis 锁确保了跨进程的幂等性。此外，`run_evaluation` 方法中检查 evaluation 状态为 `COMPLETED` 时直接返回已有结果（`evaluation_service.py:367-368`），提供了二次幂等保障。

**Q: Redis 缓存的内存增长如何控制？**

A: 当前 24h TTL（`config.py:46`）已经提供了自然淘汰机制。每天 60 万条 entry，假设每条约 5KB，峰值内存约 3GB。可以配置 Redis `maxmemory-policy=allkeys-lru` 作为安全网，并通过 Prometheus 监控 `redis_memory_used_bytes` 设置告警。
