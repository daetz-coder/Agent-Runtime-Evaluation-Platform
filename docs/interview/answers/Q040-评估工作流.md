## 问题

请画出评估工作流：`validate → 六维评估 → aggregate` 的完整流程。

## 参考答案

评估工作流定义在 `app/graphs/evaluation_graph.py` 中，核心入口是 `create_evaluation_graph()` 函数（第 374-419 行）。整体流程如下：

```
输入: task_id + goal + trajectory + context
       |
       v
+-------------------+
|  validate_input    |  检查 goal / trajectory / task_id 是否为空
|  (line 102-110)    |  缺失则设置 error 字段并终止
+--------+----------+
         | (通过验证)
         v
+-------------------+
| evaluate_planning  | --+
|  (line 113-128)    |   |
+-------------------+   |
+-------------------+   |
| evaluate_tactical  |   |
|  (line 131-146)    |   |
+-------------------+   |
+-------------------+   |
| evaluate_tool_use  |   |  在 StateGraph 中为串行链
|  (line 149-164)    |   |  (line 407-413)
+-------------------+   |  在生产环境通过 asyncio.gather 并行执行
+-------------------+   |  (line 456, evaluate_parallel)
| evaluate_memory    |   |
|  (line 167-182)    |   |
+-------------------+   |
+-------------------+   |
| evaluate_replan    |   |
|  (line 185-200)    |   |
+-------------------+   |
+-------------------+   |
| evaluate_retrieval | --+
|  (line 203-218)    |
+-------------------+
         |
         v
+--------------------+
| aggregate_results   |  读取 6 个维度分数
|   (line 221-290)    |  计算加权总分 (weighted_overall)
+--------+-----------+  生成 summary (_generate_summary, line 293)
         |               生成 recommendations (_generate_recommendations, line 321)
         v
输出: OverallEvaluation
```

**State 结构**（`EvaluationState`, 第 50-69 行）是一个 `TypedDict`，包含输入字段（task_id、goal、trajectory、context）、6 个中间分数字段（planning_score 到 retrieval_score）以及输出字段（overall_evaluation、error）。

**每个评估节点**的执行逻辑相同：(1) 实例化对应 Evaluator；(2) 调用 `_convert_trajectory_steps` 将原始 dict 转为 `TrajectoryStep` 对象（第 72-87 行）；(3) 调用 `evaluator.evaluate(goal, trajectory, context)`；(4) 将结果 `.model_dump()` 存入 state。

**aggregate_results**（第 221-290 行）读取 6 个维度分数，调用 `score_values()` 和 `weighted_overall()` 计算加权总分，然后通过 `_generate_summary()` 生成中文文本摘要（包含综合评分等级、最强/最弱维度），通过 `_generate_recommendations()` 生成改进建议（优先使用 LLM 生成的 suggestions，回退到基于阈值的硬编码建议）。

**串行 vs 并行**：StateGraph 中定义的是串行链（第 407-413 行），但 `evaluation_service.py` 第 427-448 行的 `EVAL_PARALLEL` 开关默认为 True，在生产环境中走 `evaluate_parallel()` 路径，使用 `asyncio.gather` 同时运行 6 个评估器，性能从约 71 秒降至约 15 秒。

## 代码依据

- `app/graphs/evaluation_graph.py:50-69` — EvaluationState TypedDict 定义所有状态字段
- `app/graphs/evaluation_graph.py:102-110` — validate_input 节点，校验 goal/trajectory/task_id
- `app/graphs/evaluation_graph.py:113-218` — 6 个评估节点（planning -> tactical -> tool_use -> memory -> replan -> retrieval）
- `app/graphs/evaluation_graph.py:221-290` — aggregate_results 聚合节点，计算加权总分并构建 OverallEvaluation
- `app/graphs/evaluation_graph.py:293-318` — _generate_summary，根据平均分生成中文摘要
- `app/graphs/evaluation_graph.py:321-371` — _generate_recommendations，优先 LLM 建议，回退到硬编码
- `app/graphs/evaluation_graph.py:374-419` — create_evaluation_graph，构建 StateGraph 并 .compile()
- `app/services/evaluation_service.py:427-448` — EVAL_PARALLEL 开关，控制串行/并行执行路径

## 回答要点

- StateGraph 定义了串行节点链，但生产环境默认走 evaluate_parallel() 并行路径
- 每个评估节点结构统一：实例化 Evaluator -> 转换 trajectory -> 调用 evaluate() -> 存入 state
- aggregate_results 通过 weighted_overall() 计算加权总分，权重来自配置
- summary 和 recommendations 生成有 LLM 优先 + 硬编码回退的双层策略
- 六维评估器分别为：planning、tactical、tool_use、memory、replan、retrieval

## 常见追问

**Q: 为什么 StateGraph 中用串行链而不是 LangGraph 原生的并行分支？**

A: LangGraph 的 StateGraph 使用 TypedDict 作为共享状态，并行写入同一 state 字段可能产生冲突。因此图中定义为串行链（line 407-413），但在 service 层通过 `EVAL_PARALLEL` 开关切换到 `evaluate_parallel()`（line 427-448），该函数绕过 LangGraph，直接用 `asyncio.gather` 并发调用 6 个独立的 Evaluator 实例，各自返回独立结果，最后统一聚合。

**Q: 如果某个评估节点失败了怎么办？**

A: 每个评估节点都有 try/except（如第 126-128 行），失败时返回 `{"overall": 0, "feedback": "Error: ..."}` 占位数据，不会阻断后续节点。aggregate_results 中的 `_with_defaults()` 函数（第 90-99 行）会为缺失字段填入默认值。
