## 问题

轨迹数据不完整（例如只有 tool_call 没有 tool_result）时，各 Evaluator 如何表现？

## 参考答案

在真实生产环境中，轨迹数据不完整是常态而非异常——网络超时导致 `tool_result` 丢失、Agent 进程崩溃中断轨迹记录、沙箱执行异常未被捕获等情况频繁发生。本平台的六个评估器各自独立处理缺失数据，策略不完全一致，但总体遵循"优雅降级而非崩溃"的设计原则。

**1. 各评估器的缺失数据处理策略**

不同评估器对缺失数据的处理方式可分为三类：

**返回零分 + 标记不适用（applicable=False）：**

- **ToolUseEvaluator**（`app/evaluators/tool_use_evaluator.py:99-108`）：当 `_extract_tool_calls` 返回空列表时，返回 `applicable=False`，所有子维度归零，并附带理由 `"轨迹中未包含工具调用，该维度已从综合评分中剔除。"`。
- **ReplanEvaluator**（`app/evaluators/replan_evaluator.py:111-121`）：当没有 replan 事件且 `_detect_missed_replans` 也未发现连续失败时，返回 `applicable=False`，理由为 `"Agent 顺利完成未触发重规划，该维度已从综合评分中剔除。"`。这实际上是"数据完整但场景不适用"的情况。

**返回零分（无 applicable 标记）：**

- **PlanningEvaluator**（`app/evaluators/planning_evaluator.py:100-108`）：当 `_extract_plans` 和 `_extract_plan_updates` 均为空时，返回所有子维度为 0，feedback 为 `"No planning steps found in trajectory. Agent did not create an explicit plan."`。注意：此评估器不设置 `applicable` 标记。
- **RetrievalEvaluator**（`app/evaluators/retrieval_evaluator.py:128-132`）：当 `_extract_retrievals` 未找到含 `retrieved_docs` 的步骤时，返回 `overall=0`，feedback 为 `"No retrieval steps found in trajectory."`。

**JSON 解析失败时回退到 50 分：**

所有评估器的 `_parse_scores` 方法在 JSON 解析失败时会回退到默认分数 50。例如：
- `planning_evaluator.py:219-225`：coverage/ordering/granularity/completeness 均回退到 50
- `tool_use_evaluator.py:206-211`：selection_quality/parameter_accuracy/result_utilization 均回退到 50
- `replan_evaluator.py:302-307`：trigger_appropriateness/adaptation_quality/learning_from_failure 均回退到 50
- `retrieval_evaluator.py:191-199`：relevance/evidence_accuracy/coverage 均回退到 50

回退到 50 而非 0 是一个有意的设计选择——当 LLM 返回了无法解析的 JSON 时，说明评估过程出了问题（可能是 prompt 模板问题或 LLM 输出格式异常），此时给 50 分（中性分数）比给 0 分（极端惩罚）更合理。

**2. "不适用"机制如何影响综合评分**

关键区别在于 `applicable=False` 的维度会被 `scoring.py` 中的 `weighted_overall`（`app/evaluators/scoring.py:25-44`）完全排除。该函数遍历所有维度时，会调用 `is_applicable()`（第 8-12 行）检查 `applicable` 字段：

```python
def is_applicable(dimension_result: Any) -> bool:
    if not isinstance(dimension_result, Mapping):
        return True
    return dimension_result.get("applicable", True) is not False
```

如果维度不适用，该维度的权重不会加入分母（第 38-41 行），剩余维度的权重会被归一化。例如，如果 `tool_use` 和 `replan` 都标记为不适用，综合分只会基于 planning、tactical、memory、retrieval 四个维度计算，权重总和从 1.0 重新归一化。这比简单地给 0 分更公平——一个纯计算任务没有工具调用，不应该因为 tool_use=0 而被惩罚。

**3. tool_call 缺少 tool_result 时 ToolUseEvaluator 的具体表现**

当轨迹中存在 `tool_call` 但缺少对应的 `tool_result` 时，`ToolUseEvaluator.evaluate()` 的行为如下：

- 第 96 行 `_extract_tool_calls` 能正常提取工具调用记录（`base.py:159-170` 按 `action_type == "tool_call"` 过滤）
- 第 97 行 `_extract_tool_results` 返回空列表（`base.py:198-211` 按 `action_type == "tool_result"` 过滤，找不到匹配项）
- 第 99 行 `if not tool_calls` 为 False（有 tool_calls），不会触发 early return
- 第 114 行 `execution_results_text` 回退为 `"No tool results recorded"`
- LLM 评估时会看到工具调用记录但没有执行结果，`selection_quality` 和 `parameter_accuracy` 仍可评估（基于工具名和参数），但 `result_utilization` 会因缺乏结果信息而得分偏低

**4. aggregate_results 的兜底机制**

无论评估器返回什么，`aggregate_results` 节点（`app/graphs/evaluation_graph.py:221-290`）都会通过 `_with_defaults`（第 90-99 行）填充缺失字段。该函数为每个维度定义了默认值字典（如 planning 的 `{"coverage": 0, "ordering": 0, "granularity": 0, "completeness": 0}`），确保 Pydantic 模型反序列化不会因缺失字段而崩溃。

同时，每个评估节点的 try/except 块（如 `evaluate_planning` 的第 126-128 行）会捕获所有异常，返回 `{"overall": 0, "feedback": f"Error: {str(e)}"}`，确保单个评估器的失败不会导致整个评估流程中断。

**5. 真实场景中的不完整轨迹**

常见导致轨迹不完整的原因：
- **网络超时**：Agent 调用远程工具后，tool_result 的 HTTP 回调超时，SDK 收集器的 `_http_with_retry`（`sdk/collector.py:682-730`）在 3 次重试后放弃
- **进程崩溃**：Agent 进程在工具执行期间崩溃，`on_tool_end` 回调未被触发
- **沙箱超时**：代码执行沙箱超过 `SANDBOX_TIMEOUT`（默认 30 秒，`app/core/config.py:103`）被强制终止
- **批量 flush 竞态**：SDK 收集器在批量 flush 时（`sdk/collector.py:437-446`），如果 HTTP 请求失败且缓冲区已清空，部分步骤可能丢失

## 回答要点

- 六个评估器各自独立处理缺失数据，策略不完全一致：tool_use/replan 标记 `applicable=False`，planning/retrieval 返回零分，JSON 解析失败回退到 50 分
- `applicable=False` 的维度会被 `weighted_overall` 完全排除（分子分母都不计入），比给 0 分更公平
- tool_call 缺少 tool_result 时，ToolUseEvaluator 仍可评估 selection_quality 和 parameter_accuracy，但 result_utilization 会因缺乏执行结果而得分偏低
- `_with_defaults` 函数（`evaluation_graph.py:90-99`）为所有维度填充默认字段，确保 Pydantic 反序列化不会崩溃
- 每个评估节点都有 try/except 兜底，单个评估器失败不会中断整个评估流程
- JSON 解析失败回退到 50 分（中性分数）而非 0 分，是一个有意的设计选择

## 常见追问

**Q: 为什么不把所有缺失数据都统一标记为 `applicable=False`？**

A: 因为"数据缺失"和"场景不适用"是两种不同的语义。ReplanEvaluator 标记 `applicable=False` 表示"Agent 顺利完成无需重规划"，这是合理的场景不适用。但 PlanningEvaluator 没有 plan 记录可能是 Agent 故意跳过了规划——这本身就是一个应该被评估的缺陷（给 0 分），不应被排除。区分两者能让综合评分更准确地反映 Agent 的真实能力。

**Q: 50 分的回退值会不会掩盖问题？**

A: 短期内可能，但长期来看不会。50 分只在 LLM 返回无法解析的 JSON 时触发，这种情况在生产环境中很少见（通常 <1% 的评估）。如果频繁出现，说明 prompt 模板或 LLM 配置有问题，应该在基础设施层面修复而非依赖评分回退。而且 feedback 字段会保留原始 LLM 输出（如 `planning_evaluator.py:224`：`"feedback": content`），便于排查。

**Q: 如何监控轨迹完整性？**

A: 可以通过对比 trajectory 中 `tool_call` 和 `tool_result` 的数量来计算完整性比率。如果大量 tool_call 缺少对应 tool_result，说明 Agent 运行环境或 SDK 收集器存在问题。平台的 Prometheus 指标（`app/core/metrics.py`）可以追踪评估成功率和失败原因。
