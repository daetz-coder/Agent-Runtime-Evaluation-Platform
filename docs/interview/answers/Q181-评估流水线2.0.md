## 问题

设计一个「评估流水线 2.0」：支持自定义 Evaluator 插件、动态权重、多 Judge 投票。

## 参考答案

当前评估流水线是硬编码的 6 维度结构：`EVALUATOR_CLASSES` 字典（`app/graphs/evaluation_graph.py:480-487`）将维度名映射到评估器类，权重写死在 `settings.EVAL_DIMENSION_WEIGHTS`（`app/core/config.py:136-143`），`evaluate_parallel` 函数（`evaluation_graph.py:422-477`）用 `asyncio.gather` 并发执行。这个设计适合原型，但缺乏可扩展性。以下是「评估流水线 2.0」的设计方案。

### 1. 自定义 Evaluator 插件系统

**目标**：允许用户注册自定义评估维度，无需修改核心代码。

**接口定义**：扩展 `BaseEvaluator`（`app/evaluators/base.py:32-108`）为 `EvaluatorPlugin` 协议：

```python
class EvaluatorPlugin(Protocol):
    """评估器插件协议。"""
    name: str                    # 维度名称，如 "creativity"
    display_name: str            # 显示名称，如 "创造力评估"
    version: str                 # 插件版本
    default_weight: float        # 默认权重
    sub_dimensions: dict[str, float]  # 子维度权重，如 {"originality": 0.5, "fluency": 0.5}

    async def evaluate(self, goal: str, trajectory: list[TrajectoryStep],
                       context: dict | None = None) -> EvaluationResult: ...
    def get_prompt_template(self) -> str: ...
    def validate_output(self, raw: dict) -> EvaluationResult: ...
```

**注册机制**：实现插件注册表 `EvaluatorRegistry`：

```python
class EvaluatorRegistry:
    _plugins: dict[str, type[EvaluatorPlugin]] = {}

    @classmethod
    def register(cls, name: str, plugin_class: type[EvaluatorPlugin]):
        """注册评估器插件。"""
        cls._plugins[name] = plugin_class

    @classmethod
    def discover(cls):
        """从 entry_points 和配置文件自动发现插件。"""
        # 1. Python entry_points (evaluators.plugins group)
        for ep in importlib.metadata.entry_points(group="evaluators.plugins"):
            cls.register(ep.name, ep.load())
        # 2. 配置文件指定的额外路径
        for path in settings.EVAL_PLUGIN_PATHS:
            module = importlib.import_module(path)
            if hasattr(module, "register"):
                module.register(cls)
```

内置的 6 个评估器（planning、tactical、tool_use、memory、replan、retrieval）作为内置插件自动注册，无需迁移。

**输出标准化**：每个插件必须返回 `EvaluationResult`，包含 `overall` 分数、`sub_scores` 子维度字典、`feedback` 文本、`evidence` 引用列表。`BaseEvaluator._invoke_llm_cached`（`app/evaluators/base.py:294-372`）的缓存机制自动适用于所有插件，因为 cache key 包含了 `type(self).__name__`（`base.py:317`）。

### 2. 动态权重系统

**问题**：当前权重是静态的 `dict`（`config.py:136-143`），写在 `.env` 文件中，修改需要重启服务。

**设计方案**：

- **存储层**：将权重从 `config.py` 迁移到数据库表 `evaluation_weight_profiles`，字段包括 `profile_name`、`dimension`、`weight`、`updated_at`。保留 `config.py` 中的默认值作为 fallback
- **API 层**：新增 `PUT /api/v1/evaluations/weights` 端点，支持运行时更新权重，无需重启
- **Per-task 覆盖**：在 `EvaluationRequest` schema 中增加可选字段 `weight_overrides: dict[str, float]`，允许单次评估跳过某些维度（如 Memory 评估对纯代码生成任务无意义）
- **权重校验**：服务端校验权重值范围（0-1）并在聚合前自动归一化。`aggregate_results`（`evaluation_graph.py:221-290`）中的 `weighted_overall` 调用改为从数据库/请求上下文获取权重，而非直接读 `settings`
- **Profile 管理**：支持多组权重配置（"balanced"、"planning-focused"、"tool-focused"），用户可以按场景切换

**缓存策略**：权重变更频率低，使用 Redis 缓存 5 分钟，避免每次评估都查库。

### 3. 多 Judge 投票机制

**现状**：项目已有 `ConsensusEvaluator`（`app/evaluators/consensus.py:64-69`），实现了多模型独立评分并输出均值和标准差。它支持跨厂商共识（DeepSeek + OpenAI + Anthropic）、同厂商多模型共识（deepseek-chat + deepseek-reasoner）、温度多样性共识三种策略（`consensus.py:42-61`）。

**2.0 设计**：将共识机制泛化为通用的多 Judge 投票框架：

```python
class VotingStrategy(Enum):
    MAJORITY = "majority"           # 多数投票
    AVERAGE = "average"             # 均值
    TRIMMED_MEAN = "trimmed_mean"   # 去掉最高最低后均值
    WEIGHTED = "weighted"           # 按历史准确率加权

class MultiJudgeAggregator:
    def __init__(self, strategy: VotingStrategy = VotingStrategy.AVERAGE):
        self.strategy = strategy

    async def aggregate(self, judge_results: list[JudgeResult]) -> AggregatedScore:
        scores = [r.score for r in judge_results]
        if self.strategy == VotingStrategy.TRIMMED_MEAN:
            scores = sorted(scores)[1:-1]  # 去掉最高最低
        # ...
```

**关键增强**：

- **置信度加权**：为每个 Judge 维护历史准确率（与人类标注的偏差），准确率高的 Judge 权重更大
- **异常值检测**：当某个 Judge 的评分与其他 Judge 偏差超过 2 个标准差时，自动标记为 outlier 并降低其权重
- **并行执行**：复用 `evaluate_parallel` 的 `asyncio.gather` 模式（`evaluation_graph.py:456`），N 个 Judge 并发调用
- **透明度**：保留 `_judge_raw` 数据（`base.py:39-40`），每个 Judge 的原始 prompt 和 response 都可追溯

### 4. 流水线架构

完整的 Pipeline 流程：

```
Plugin Loader → Weight Resolver → Judge Router → Parallel Executor → Vote Aggregator → Score Normalizer
```

- **Plugin Loader**：从 `EvaluatorRegistry` 加载本次评估需要的插件列表
- **Weight Resolver**：从数据库/请求上下文获取权重配置，校验并归一化
- **Judge Router**：根据配置决定每个维度使用几个 Judge、哪些 LLM 模型
- **Parallel Executor**：`asyncio.gather` 并发执行所有评估任务（维度 x Judge 矩阵）
- **Vote Aggregator**：对同一维度的多个 Judge 结果应用投票策略
- **Score Normalizer**：计算加权总分，生成 summary 和 recommendations

**与现有代码的兼容**：`evaluate_parallel`（`evaluation_graph.py:422-477`）重构为调用 Pipeline 的入口。`EVALUATOR_CLASSES` 字典替换为 `EvaluatorRegistry.get_active()`。`aggregate_results`（`evaluation_graph.py:221-290`）中的权重读取改为从 Pipeline context 获取。

### 5. 向后兼容

- 现有 6 个评估器自动注册为内置插件，`EVALUATOR_CLASSES` 字典保留为只读兼容层
- `config.py` 中的 `EVAL_DIMENSION_WEIGHTS` 作为默认值，数据库无配置时 fallback 到此值
- API 响应格式不变，新增字段为可选
- `ConsensusEvaluator`（`consensus.py`）作为 `MultiJudgeAggregator` 的特殊实现保留

## 回答要点

- 插件系统的核心是 `EvaluatorPlugin` 协议 + `EvaluatorRegistry` 注册表，内置 6 评估器无需迁移
- 动态权重需要从 config 迁移到数据库，支持 per-task 覆盖和自动归一化
- `ConsensusEvaluator` 已经实现了多 Judge 的原型，2.0 需要泛化投票策略和置信度加权
- Pipeline 的 6 个阶段（Loader → Resolver → Router → Executor → Aggregator → Normalizer）取代当前的硬编码 `evaluate_parallel`
- cache key 中的 `type(self).__name__`（`base.py:317`）天然支持多插件缓存隔离
- 所有变更保持向后兼容，现有 API 响应格式不变

## 常见追问

**Q: 如何防止恶意插件拖垮系统？**

A: 三层防护：(1) 插件必须继承 `BaseEvaluator` 并实现 `evaluate` 方法，否则注册失败；(2) 每个插件调用 LLM 时受 `_invoke_llm_cached` 的统一缓存和超时控制；(3) 在 `evaluate_parallel` 中为每个插件设置独立的 `asyncio.wait_for` 超时（默认 30s），超时则返回零分并告警。

**Q: 动态权重如何处理维度增减？**

A: 新增维度通过插件注册自动生效，权重默认值由插件的 `default_weight` 属性决定。删除维度只需从权重配置中移除，`aggregate_results` 中的 `weighted_overall` 会自动忽略不在权重表中的维度分数。归一化确保总权重始终为 1.0。

**Q: 多 Judge 投票的性能开销如何控制？**

A: 假设 6 个维度 x 3 个 Judge = 18 次 LLM 调用。通过 `asyncio.gather` 并发执行，墙钟时间与单 Judge 相同（取决于最慢的一次调用）。LLM 缓存命中时（`base.py:328`），同一 trajectory 的重复评估几乎零成本。对于高吞吐场景，可以只对关键维度（planning、tactical）启用多 Judge，其他维度单 Judge 即可。
