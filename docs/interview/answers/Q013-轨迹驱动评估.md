## 问题

请解释「轨迹驱动评估（Trajectory-driven Evaluation）」：平台本身不运行被评 Agent，这种设计有什么优缺点？

## 参考答案

轨迹驱动评估的核心思想是：平台接收一段预先录制好的 agent 动作序列（trajectory），而非 agent 本身。评估器对这段轨迹进行离线打分，而不是在 agent 运行时实时干预。

**轨迹的结构。** 每条轨迹由若干 `TrajectoryStep` 组成，每个 step 包含 `step_number`、`action_type`、`action_detail`、`observation` 等字段（参见 `app/evaluators/base.py:128-137` 的 `_format_trajectory_raw`）。`action_type` 有 14 种常量（`app/models/action_types.py:8-57`），覆盖规划（PLAN）、工具调用（TOOL_CALL）、思考（THINK）、重规划（REPLAN）、检索（RETRIEVAL）等维度。

**收集机制。** `sdk/collector.py:148-798` 中的 `TrajectoryCollector` 是收集器的核心。它是一个线程安全的单例，通过 `ContextVar` 隔离不同并发会话（line 145-152）。外部 agent 通过 SDK 的 adapter（LangGraph adapter / LLM Proxy / Callback）将动作记录到 collector，collector 在 `finish_async` 时批量发送到平台。

**优点：**
1. **框架无关。** 只要 agent 能通过 SDK adapter 输出符合 schema 的轨迹，无论底层用 LangGraph、LangChain 还是自研框架，都可以被评估。
2. **无运行时耦合。** 平台不需要知道 agent 如何推理，只关心记录下来的动作序列。这降低了集成复杂度。
3. **可重复评估。** 同一条轨迹可以用不同版本的评估器重新打分，适合评估器迭代。
4. **支持离线/批量评估。** 轨迹可以先存储，后续批量触发评估（`collector.finish(auto_run=True)` 中的 fire-and-forget 模式）。

**缺点：**
1. **可观测性依赖埋点质量。** 如果 agent 没有正确埋点，关键动作（如中间推理、工具选择决策）会丢失，评估器只能看到部分轨迹。
2. **无法交互。** 平台不能在 agent 执行中途向其提问或修改其行为，只能事后分析。
3. **Schema 需要约定。** 轨迹的 `action_type` 和字段结构需要 agent 开发者与平台对齐。

**Sandbox 例外。** 平台也支持 Sandbox 模式——在 Docker 容器中实际运行 agent。但即使在 Sandbox 模式下，评估仍然基于捕获到的轨迹进行，而非实时干预 agent 的推理过程。

## 代码依据

- `sdk/collector.py` — TrajectoryCollector 单例，统一收集所有 agent 的轨迹
- `sdk/collector.py` — `start_async` 创建任务并开始记录
- `sdk/collector.py` — `finish_async` 将轨迹持久化并触发评估
- `sdk/schemas.py` — 14 种 ActionType 的 Pydantic Schema 定义
- `app/evaluators/base.py` — `_format_trajectory` 将轨迹转为可读文本
- `app/models/action_types.py` — re-export ActionType 和 Schema

## 回答要点

- 轨迹驱动 = 平台接收预录制的动作序列，不运行 agent 本身
- 优点：框架无关、无运行时耦合、可重复评估、支持离线批量
- 缺点：可观测性依赖埋点质量、无法运行时交互、需约定 schema
- Sandbox 模式是例外，但评估仍基于捕获的轨迹
- TrajectoryCollector 是核心收集器，通过 ContextVar 实现并发隔离

## 常见追问

**Q: 如果 agent 没有埋点怎么办？**

A: 平台会收到一条几乎为空的轨迹，评估分数自然很低。这本身也是一种信号——说明 agent 缺乏可观测性。我们在 SDK 中提供了三种 adapter（参见 Q030），就是为了降低埋点门槛。

**Q: 能不能在 agent 运行时实时评估？**

A: 当前架构不支持。实时评估需要 agent 主动向平台流式推送每一步动作，技术上可行（collector 的 `record` 方法是同步的），但评估器（如 LLM-as-Judge）有延迟，不适合实时反馈。
