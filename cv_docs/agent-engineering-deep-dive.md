## 第一部分：全景视角 — Agent 工程的三维模型

### 1.1 为什么需要多维度理解？

很多人把 "Agent Engineering" 等同于 "选一个推理策略（如 ReAct）"，但实际上构建一个生产级 Agent 系统涉及多个正交的工程维度。一个 Agent 用 ReAct 策略，运行在精心设计的 Harness 中，通过 Workflow 编排多步任务 —— 这三维缺一不可。

**一个常见误区**: 把 Workflow、Harness、Loop 理解为"上下堆叠的三层"。实际上：

- **Loop 不是独立层** — 它是 Harness 的核心运转机制，就像心跳是心脏的一部分
- **Workflow 和 Harness 不是上下层** — 它们是正交的编织关系，互为表里

### 1.2 三维架构总览

```
┌──────────────────────────────────────────────────────────────┐
│                    用户请求 / 外部事件                         │
└───────────────────────────┬──────────────────────────────────┘
                            ↓
    ┌───────────────────────────────────────────────────────┐
    │         策略层 (推理决策层)                              │
    │  "Agent 每一步用什么方式思考和行动？"                     │
    │  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
    │  │  ReAct   │  │ Plan&Exec│  │ Reflexion│  ...        │
    │  └──────────┘  └──────────┘  └──────────┘             │
    └───────────────────────────┬───────────────────────────┘
                                │ 被使用
                                ↓
┌═══════════════════════════════════════════════════════════════┐
║                                                               ║
║   Workflow ←────────── 编织 ──────────→ Harness               ║
║   (结构编排)                             (运行时)              ║
║                                                               ║
║   · DAG / 状态机 / Pipeline             · System Prompt       ║
║   · 任务依赖与顺序                       · Tools + Perms      ║
║   · 并行与条件分支                       · Context Mgmt       ║
║   · 错误处理与重试           ┌────────────────────────────┐   ║
║                              │  Loop 机制 (Harness 的心脏) │   ║
║   关注: 多个 Agent           │  · 终止条件  · 步数控制     │   ║
║   怎么组合                   │  · 预算控制  · 停滞检测     │   ║
║                              │  · 反思改进  · 人机协同     │   ║
║                              └────────────────────────────┘   ║
║                                                               ║
╚═══════════════════════════════════════════════════════════════╝
                            ↓
                    执行结果 / 输出
```

### 1.3 Workflow 和 Harness 的关系：编织，不是分层

**关键洞察**: Workflow 和 Harness 是同一系统的两个正交视角，不是上下堆叠的层级。

```
类比: 导演 (Workflow) 和 片场 (Harness)

导演 (Workflow):                    片场 (Harness):
  · 决定先拍什么后拍什么               · 提供灯光、道具、摄影机
  · 决定哪些场景并行拍                 · 提供演员表演的环境
  · 决定什么时候需要补拍               · 提供安全保障和规则

它们不是上下级关系，而是:
  导演在片场里工作
  片场支撑导演的调度
```

**三种典型关系模式**:

```
模式 1: Workflow 调用 Harness (最常见)
┌─────────────────────────────────────────────────┐
│  Workflow                                       │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    │
│  │ Step 1  │───→│ Step 2  │───→│ Step 3  │    │
│  └────┬────┘    └────┬────┘    └────┬────┘    │
│       ↓              ↓              ↓          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    │
│  │Harness A│    │Harness B│    │Harness C│    │
│  └─────────┘    └─────────┘    └─────────┘    │
└─────────────────────────────────────────────────┘

模式 2: Harness 内部有 Workflow (Agent 内部编排)
┌─────────────────────────────────────────────────┐
│  Harness (Agent 的运行时)                        │
│  ┌───────────────────────────────────────────┐  │
│  │  Agent 内部的子任务 Workflow               │  │
│  │  ┌──────┐   ┌──────┐   ┌──────┐         │  │
│  │  │ 搜索  │──→│ 分析  │──→│ 写报告 │        │  │
│  │  └──────┘   └──────┘   └──────┘         │  │
│  └───────────────────────────────────────────┘  │
│  Loop + Tools + Context                         │
└─────────────────────────────────────────────────┘

模式 3: 互相嵌套
┌─────────────────────────────────────────────────────────┐
│  Workflow A (顶层)                                       │
│  ┌──────────────────┐       ┌──────────────────┐        │
│  │ Harness (Agent1) │       │ Harness (Agent2) │        │
│  │ ┌──────────────┐ │       │ ┌──────────────┐ │        │
│  │ │ Workflow B    │ │       │ │ Workflow C    │ │        │
│  │ └──────────────┘ │       │ └──────────────┘ │        │
│  └──────────────────┘       └──────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

### 1.4 Loop 为什么属于 Harness？

Loop 是 Harness 的核心运转机制，不是独立层。当你定义一个 Harness 时，你同时定义了：

| Harness 组件 | 实际上就是 Loop 的... |
|---|---|
| 工具注册 | 循环中"可以做什么" |
| 权限控制 | 循环中"什么需要确认" |
| 上下文管理 | 循环中"记住什么" |
| 终止条件 | 循环中"什么时候停" |
| 步数/预算限制 | 循环中"能跑多久" |

它们是同一个事物的不同侧面，不应该分层。

### 1.5 三维的核心区别

| 维度 | Workflow 工程 | Harness 工程 (含 Loop) | 策略层 |
|------|:---:|:---:|:---:|
| **核心问题** | 多个 Agent 怎么组合？ | 单个 Agent 怎么跑？ | 每一步怎么想？ |
| **类比** | 导演的分镜脚本 | 片场 + 拍摄节奏 | 演员的表演方法 |
| **关注点** | 结构、顺序、依赖 | 环境、循环、终止 | 推理、规划、行动 |
| **变化频率** | 任务级别变化 | 平台级别稳定 | 策略级别选择 |
| **典型工件** | DAG 定义、状态图 | System Prompt、工具 Schema、Loop Config | ReAct Prompt、Plan Template |
| **作用对象** | 多个任务/Agent 的组合 | 单个 Agent 的完整运行时 | 单个 LLM 调用的决策 |

## 第二部分：策略层 — Agent 的推理与决策方法

> 策略层是大家最熟悉的部分，决定 Agent "如何思考"。以下按演化脉络详细展开。

### 2.1 Chain of Thought (CoT)

**论文**: Wei et al., 2022

**核心思想**: 引导模型逐步推理，将复杂问题分解为中间步骤。

**三种变体对比**:

| 变体 | 触发方式 | 推理示例来源 | 优势 | 劣势 |
|------|----------|-------------|------|------|
| Zero-shot CoT | "Let's think step by step" | 模型自生成 | 无需示例 | 质量不稳定 |
| Few-shot CoT | 提供带推理的示例 | 人工编写 | 质量可控 | 需要领域专家 |
| Auto-CoT | 自动生成推理示例 | 模型生成 + 聚类 | 自动化 | 可能引入噪声 |
| Self-Consistency | 多路径采样 + 投票 | 多次生成 | 准确率高 | 成本 N 倍 |

**Self-Consistency 工作原理**:

```
问题 → 路径1: 步骤A→B→C→答案X (概率0.3)
     → 路径2: 步骤D→E→F→答案X (概率0.4)  ──投票──→ 答案X
     → 路径3: 步骤G→H→I→答案Y (概率0.3)
```

**局限**: 纯推理，无法与外部世界交互；推理链中的错误会级联传播。

---

### 2.2 ReAct (Reasoning + Acting)

**论文**: Yao et al., 2022

**核心思想**: 推理与行动交替进行，形成 Thought → Action → Observation 循环。

**详细执行流程**:

```
┌──────────────────────────────────────────────────────┐
│  用户: "2024年诺贝尔物理学奖得主的研究领域是什么？"      │
└──────────────────────────┬───────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────┐
│  Thought: 我需要查找2024年诺贝尔物理学奖的信息          │
│  Action:  search("2024 Nobel Prize Physics winner")   │
│  Observation: Geoffrey Hinton 和 John Hopfield        │
└──────────────────────────┬───────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────┐
│  Thought: 他们因机器学习和人工神经网络方面的发现获奖     │
│  Action:  finish("机器学习与人工神经网络")               │
└──────────────────────────────────────────────────────┘
```

**ReAct 的隐含缺陷 — 缺乏全局视野**:

ReAct 是贪心式的：每一步只看当前最优，不考虑全局。当任务需要 "先做 A 才能做好 B" 时，ReAct 可能直接去做 B 导致质量低下。

### 2.3 Plan-and-Execute

**核心改进**: 将规划与执行分离，先有全局计划，再逐步执行。

**三种规划策略对比**:

| 策略 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **一次性规划** | 开始时生成完整计划，不调整 | 简单、快速 | 无法应对意外 |
| **滚动式规划** | 每执行一步后重新规划 | 适应性强 | Token 消耗大 |
| **混合式规划** | 初始计划 + 条件触发 Re-plan | 平衡灵活性和效率 | 实现复杂度中等 |

**Re-plan 触发条件**:

```
触发 Re-plan 的情况:
├── 执行步骤失败（工具调用出错、返回异常）
├── 执行结果与预期不符（信息不匹配）
├── 发现新的约束条件（用户补充需求）
└── 执行时间/成本超出预算

不触发 Re-plan 的情况:
├── 步骤成功执行，结果符合预期
├── 仅有微小偏差，不影响后续步骤
└── 剩余步骤足够简单，无需调整
```

### 2.4 Reflexion (自我反思)

**论文**: Shinn et al., 2023

**核心创新**: 引入跨尝试的反思记忆，从失败中学习。

**Reflexion 与传统重试的本质区别**:

```
传统重试:
  尝试1 → 失败 → 尝试2 (和尝试1完全一样) → 失败 → ...
  本质上是"盲目重试"

Reflexion:
  尝试1 → 失败 → 反思"我在X步骤用了错误的参数，应该用Y"
  尝试2 → 使用反思记忆 → 避免同样的错误 → 可能成功
  本质上是"经验学习"
```

**反思质量的关键因素**:

| 因素 | 好的反思 | 差的反思 |
|------|---------|---------|
| **具体性** | "在第3步，我用 `sort(reverse=True)` 导致降序排列，应该用默认升序" | "排序有问题" |
| **可操作性** | "下次应该先验证输入数据的格式再处理" | "需要更仔细" |
| **归因准确** | "失败是因为工具返回了空列表，我应该检查空值" | "可能是模型的问题" |

---

### 2.5 Tree of Thoughts (ToT)

**论文**: Yao et al., 2023

**核心思想**: 从线性推理扩展为树状搜索，支持多路径探索和回溯。

**搜索策略详细对比**:

| 策略 | 原理 | 适用场景 | Token 成本 |
|------|------|---------|-----------|
| **BFS** | 探索所有节点的当前层 | 解空间宽、需要全面探索 | 高 |
| **DFS** | 深入最有前途的路径 | 解空间深、需要深度探索 | 中 |
| **Beam Search** | 维护 top-k 候选路径 | 平衡探索与利用 | 可控 |

**ToT vs CoT 决策树**:

```
问题是否需要探索多种可能性？
├─ 否 → 用 CoT
└─ 是 → 解空间有多大？
    ├─ 小（<5种）→ CoT + Self-Consistency
    ├─ 中（5-20种）→ ToT + Beam Search
    └─ 大（>20种）→ ToT + BFS/DFS + 剪枝
```

---

### 2.6 LATS (Language Agent Tree Search)

**论文**: Zhou et al., 2023

**核心创新**: 将 MCTS（蒙特卡洛树搜索）与 LLM Agent 统一。

**LATS 的四阶段循环**:

```
1. Selection（选择）
   从根节点开始，用 UCB1 公式选择最优子节点
   平衡 "探索未尝试的路径" 和 "深入已知好路径"

2. Expansion（扩展）
   在选中的叶节点生成新的候选行动
   LLM 提供多个可能的下一步

3. Simulation（模拟）
   从新节点开始，用 LLM 模拟执行到终点
   评估这条路径的最终价值

4. Backpropagation（回溯）
   将模拟结果沿路径回传，更新每个节点的价值估计
```

**LATS 相对于 ToT 的关键改进**:

| 维度 | ToT | LATS |
|------|-----|------|
| 评估方式 | LLM 直接打分 | MCTS 模拟 + LLM 评估 |
| 评估质量 | 依赖 LLM 的判断力 | 通过多次模拟更准确 |
| 行动能力 | 仅推理 | 推理 + 行动（可调用工具） |
| 探索策略 | BFS/DFS/Beam | UCB1（更科学的探索-利用平衡） |

### 2.7 Multi-Agent Systems

**三种架构模式详解**:

#### 模式一：层级式（Hierarchical）

```
                  ┌─────────────┐
                  │   Manager    │  ← 分解任务、分配给子Agent、汇总结果
                  └──────┬──────┘
            ┌────────────┼────────────┐
            ↓            ↓            ↓
     ┌──────────┐ ┌──────────┐ ┌──────────┐
     │ Researcher│ │  Coder   │ │ Reviewer │  ← 各司其职
     └──────────┘ └──────────┘ └──────────┘
```

**优点**: 职责清晰、易于管理  
**缺点**: Manager 是单点瓶颈、通信开销大

#### 模式二：对等式（Peer-to-Peer）

```
     Agent A ←──────→ Agent B
       ↑    ╲        ╱    ↑
       │      ╲    ╱      │
       │        ╳         │
       │      ╱    ╲      │
       ↓    ╱        ╲    ↓
     Agent C ←──────→ Agent D
```

**优点**: 无单点故障、高度灵活  
**缺点**: 协调困难、可能死锁

#### 模式三：辩论式（Debate）

```
     ┌─────────┐    论点     ┌─────────┐
     │ Agent A  │───────────→│ Agent B  │
     └────┬────┘←───────────└────┬────┘
          │        反驳           │
          └──────────┬───────────┘
                     ↓
              ┌─────────────┐
              │  Moderator   │  ← 综合双方观点，做出最终裁决
              └─────────────┘
```

**优点**: 减少幻觉、提高准确性  
**缺点**: 成本高、不适合所有任务

**Multi-Agent 通信协议**:

| 协议 | 描述 | 实现复杂度 | 适用场景 |
|------|------|:---:|------|
| **消息传递** | Agent 直接发送消息给特定 Agent | 低 | 简单协作 |
| **共享黑板** | 所有 Agent 读写公共状态空间 | 中 | 需要全局协调 |
| **发布-订阅** | Agent 发布事件，订阅者自动响应 | 中 | 事件驱动系统 |
| **请求-响应** | 同步调用，等待返回 | 低 | 工具调用式 |
| **流式管道** | 输出直接流入下一个 Agent 的输入 | 低 | 数据处理流水线 |

## 第三部分：Workflow 工程 — 任务编排与流程控制

> Workflow 工程回答的问题是：**多个 Agent、多个步骤之间，以什么结构和顺序协作？**

### 3.1 什么是 Workflow 工程？

Workflow 工程是 Agent 系统的 **编排层**，定义了：
- 任务如何分解为子任务
- 子任务之间的依赖关系
- 执行顺序（串行/并行/条件分支）
- 错误处理和重试策略
- 状态管理和持久化

**与策略层的关键区别**:

| 维度 | 策略层 | Workflow 层 |
|------|--------|------------|
| **关注点** | 单个 Agent 如何推理 | 多个步骤如何编排 |
| **粒度** | 一次 LLM 调用的决策 | 整个任务的执行流程 |
| **类比** | 棋手的下棋策略 | 比赛的赛程安排 |
| **变化频率** | 随任务类型变化 | 相对稳定 |

### 3.2 四种核心 Workflow 模式

#### 模式一：DAG（有向无环图）

```
         ┌──────┐
         │ 开始  │
         └──┬───┘
      ┌─────┴─────┐
      ↓           ↓
  ┌──────┐   ┌──────┐
  │ 步骤A │   │ 步骤B │    ← 可并行执行
  └──┬───┘   └──┬───┘
      └─────┬─────┘
           ↓
       ┌──────┐
       │ 步骤C │           ← 等待 A、B 都完成
       └──┬───┘
           ↓
       ┌──────┐
       │ 结束  │
       └──────┘
```

**特点**: 
- 节点代表任务，边代表依赖
- 无循环，适合确定性流程
- 天然支持并行执行

**适用场景**: 数据处理 Pipeline、ETL 流程、编译构建

#### 模式二：状态机（State Machine）

```
                  ┌─────────────┐
          ┌──────→│   空闲状态   │←──────┐
          │       └──────┬──────┘       │
          │              │ 收到任务      │ 任务完成
          │              ↓              │
          │       ┌─────────────┐       │
          │       │   规划中     │───────┘
          │       └──────┬──────┘
          │              │ 计划生成
          │              ↓
          │       ┌─────────────┐
     错误  │       │   执行中     │
     重试  │       └──────┬──────┘
          │              │ 执行完成/出错
          │              ↓
          │       ┌─────────────┐
          └───────│   评估中     │
                  └─────────────┘
```

**特点**:
- 明确的状态定义和转换条件
- 支持循环和回退
- 状态可持久化，支持中断恢复

**适用场景**: 客服对话流程、审批工作流、交互式任务

#### 模式三：Pipeline（流水线）

```
输入 → [阶段1: 解析] → [阶段2: 处理] → [阶段3: 验证] → [阶段4: 输出] → 结果
         ↓                ↓                ↓                ↓
       Agent A          Agent B          Agent C          Agent D
```

**特点**:
- 数据单向流动
- 每个阶段是独立的处理单元
- 支持流式处理（边处理边传递）

**适用场景**: 文档处理、内容审核、数据转换

#### 模式四：事件驱动（Event-Driven）

```
                    ┌─────────────┐
                    │  事件总线     │
                    └──────┬──────┘
          ┌────────────────┼────────────────┐
          ↓                ↓                ↓
   ┌──────────┐     ┌──────────┐     ┌──────────┐
   │ Agent A  │     │ Agent B  │     │ Agent C  │
   │ 订阅: 新任务│     │ 订阅: 数据就绪│     │ 订阅: 完成事件│
   └──────────┘     └──────────┘     └──────────┘
```

**特点**:
- 松耦合，Agent 之间不直接通信
- 通过事件触发，高度灵活
- 易于扩展新的 Agent

**适用场景**: 微服务架构、异步处理、复杂业务流程

### 3.3 Workflow 模式对比

| 维度 | DAG | 状态机 | Pipeline | 事件驱动 |
|------|:---:|:---:|:---:|:---:|
| **结构** | 有向无环图 | 状态+转换 | 线性流水线 | 事件网络 |
| **循环支持** | ✗ | ✓ | ✗ | ✓ |
| **并行执行** | ✓ | 有限 | 阶段内可并行 | ✓ |
| **复杂度** | 中 | 中-高 | 低 | 高 |
| **可预测性** | 高 | 中 | 高 | 低 |
| **灵活性** | 中 | 高 | 低 | 最高 |
| **调试难度** | 低 | 中 | 低 | 高 |
| **典型工具** | Airflow, Prefect | LangGraph, XState | Logstash, Vector | Kafka, Temporal |

### 3.4 Workflow 工程中的关键设计模式

#### 模式 1: Fan-out / Fan-in（扇出/扇入）

```
              ┌──→ Agent A ──┐
输入 → 分发器 ─┼──→ Agent B ──┼→ 汇总器 → 输出
              └──→ Agent C ──┘
```

将任务分发给多个 Agent 并行处理，然后汇总结果。

#### 模式 2: Router（路由）

```
输入 → 分类器 ─┬──→ 路径A (简单问题 → 快速Agent)
              ├──→ 路径B (复杂问题 → 深度Agent)
              └──→ 路径C (专业问题 → 专家Agent)
```

根据输入特征选择不同的处理路径。

#### 模式 3: Validator Loop（验证循环）

```
生成 → 验证 ──┬── 通过 → 输出
              └── 失败 → 反馈给生成器 → 重新生成
```

生成-验证循环，直到结果满足质量要求。

#### 模式 4: Human-in-the-Loop（人机协同）

```
Agent 处理 → 置信度检查 ──┬── 高置信度 → 自动执行
                          └── 低置信度 → 人工审核 → 继续/修正
```

在关键节点引入人工审核，平衡自动化和可靠性。

### 3.5 工程框架中的 Workflow 实现

| 框架 | Workflow 模式 | 核心抽象 | 特色 |
|------|:---:|---------|------|
| **LangGraph** | 状态机 | Graph + Node + Edge | 原生支持循环和条件分支 |
| **Temporal** | 状态机 + DAG | Workflow + Activity | 持久化、故障恢复、超时处理 |
| **Airflow** | DAG | DAG + Task + Operator | 调度、监控、重试 |
| **Prefect** | DAG | Flow + Task | 动态任务、参数化 |
| **Argo Workflows** | DAG + Pipeline | Workflow Template | K8s 原生、容器化 |
| **Dapr Workflows** | 状态机 | Workflow + Activity | 微服务编排 |

## 第四部分：Harness 工程 — Agent 运行时与循环控制

> Harness 工程回答的问题是：**单个 Agent 如何运行？它的执行循环如何运转和终止？**
>
> Harness 是 Agent 的完整运行时环境，**Loop（执行循环）是 Harness 的心脏**。

### 4.1 什么是 Harness？

**Harness**（运行时环境/框架）是包裹在 LLM 外面的一层软件基础设施。它不仅提供静态的工具和提示，还驱动 Agent 的 **执行循环** —— 这是 Harness 最核心却最容易被忽视的职责。

**Harness = 静态环境 + 动态循环**

```
┌─────────────────────────────────────────────────────────────────┐
│                         Harness                                  │
│                                                                  │
│  静态层 (环境配置):                    动态层 (循环机制):         │
│  ┌────────────────────────┐          ┌────────────────────────┐  │
│  │ · System Prompt        │          │ · while 循环驱动        │  │
│  │ · 工具定义与注册        │          │ · 终止条件判断          │  │
│  │ · 权限策略              │   驱动   │ · 步数/预算控制         │  │
│  │ · 上下文管理规则        │ ───────→ │ · 停滞检测              │  │
│  │ · 输出解析规则          │          │ · 反思与重试            │  │
│  │ · 可观测性配置          │          │ · 人机协同注入          │  │
│  └────────────────────────┘          └────────────────────────┘  │
│         "提供什么环境"                    "怎么跑、何时停"        │
└─────────────────────────────────────────────────────────────────┘
```

**一个关键洞察**: 很多人以为 Agent 的核心是 LLM，但实际上 Harness 的设计质量往往决定了 Agent 的可靠性。同一个 LLM，在不同的 Harness 中表现可能天差地别。

### 4.3 Harness 的 Loop 引擎详解

Loop 引擎是 Harness 最核心的组件 —— 它驱动 Agent 的整个执行过程。

**一个 Agent 的核心就是这个循环**:

```
while (未满足终止条件) {
    注入上下文 → 调用 LLM → 解析输出 → 执行动作 → 检查状态
}
```

看似简单，但 "终止条件" 的定义、"上下文" 的管理、"检查" 的标准，都是工程设计的核心问题。

#### 4.3.1 五种核心 Loop 模式

**模式 1: Fixed-Step Loop（固定步数循环）**

```python
for step in range(MAX_STEPS):
    action = agent.think(state)
    result = environment.execute(action)
    state = result
    if is_goal_reached(state):
        break
```

**模式 2: Goal-Based Loop（目标驱动循环）**

```python
while True:
    action = agent.think(state)
    result = environment.execute(action)
    state = result
    if goal_checker.evaluate(state):
        return SUCCESS
    if stagnation_detector.is_stuck(state_history):
        return NEED_HELP
```

**模式 3: Budget-Controlled Loop（预算控制循环）**

```python
budget = Budget(total_tokens=100000, max_time=300)
while budget.remaining() > 0:
    action = agent.think(state)
    cost = estimate_cost(action)
    if budget.remaining() < cost:
        break  # 预算不足，优雅退出
    result = environment.execute(action)
    budget.spend(cost)
    state = result
```

**模式 4: Confidence-Based Loop（置信度循环）**

```python
while True:
    action = agent.think(state)
    confidence = agent.estimate_confidence()
    if confidence >= THRESHOLD:
        return action
    if confidence < LOW_THRESHOLD:
        action = ask_human_for_help(state)
    result = environment.execute(action)
    state = result
```

**模式 5: Reflect-and-Improve Loop（反思改进循环）**

```python
for attempt in range(MAX_ATTEMPTS):
    result = agent.execute(task)
    if success_checker(result):
        return result
    reflection = reflector.generate(task, result, failure_reason)
    memory.store(reflection)
    # 下次尝试时，反思记忆会被注入上下文
```

#### 4.3.2 Loop 模式对比

| 模式 | 终止条件 | 成本可控性 | 完成质量 | 适用场景 |
|------|---------|:---:|:---:|------|
| Fixed-Step | 步数上限 | ★★★ | ★ | 快速原型、成本敏感 |
| Goal-Based | 目标达成 | ★ | ★★★ | 明确目标的任务 |
| Budget-Controlled | 预算耗尽 | ★★★ | ★★ | 按量计费、成本可控 |
| Confidence-Based | 置信度阈值 | ★★ | ★★★ | 高风险、人机协同 |
| Reflect-and-Improve | 成功或最大尝试 | ★ | ★★★ | 代码调试、探索性任务 |

#### 4.3.3 多层终止条件设计

一个健壮的 Loop 需要多层终止条件：

```
┌─────────────────────────────────────────────────────────────┐
│                    多层终止条件                                │
│                                                              │
│  第1层: 成功终止                                              │
│  ├── 目标达成 (goal_checker 返回 true)                       │
│  ├── Agent 主动声明完成 (finish action)                      │
│  └── 输出通过验证 (validator 返回 pass)                      │
│                                                              │
│  第2层: 资源耗尽                                              │
│  ├── Token 预算用尽                                          │
│  ├── 时间限制到达                                            │
│  ├── API 调用次数超限                                        │
│  └── 内存/存储不足                                           │
│                                                              │
│  第3层: 异常终止                                              │
│  ├── 连续 N 次相同错误 (陷入死循环)                           │
│  ├── 工具调用连续失败                                        │
│  ├── 输出质量持续下降                                        │
│  └── 安全策略触发                                            │
│                                                              │
│  第4层: 人工干预                                              │
│  ├── 用户主动取消                                            │
│  ├── 低置信度请求人工决策                                    │
│  └── 超出 Agent 能力范围                                     │
└─────────────────────────────────────────────────────────────┘
```

#### 4.3.4 Loop 工程中的关键问题

**问题 1: 如何检测 Agent 陷入死循环？**

```
检测策略:
├── 行动重复检测: 最近 N 步的行动完全相同
├── 状态停滞检测: 最近 N 步状态没有实质变化
├── 输出相似度: 最近 N 次输出过于相似
└── 资源消耗异常: Token 消耗速度异常高但无进展
```

**问题 2: 如何在循环中注入人类反馈？**

```
          ┌──────────────────────────────────────┐
          │           Agent 执行循环               │
          │  ┌──────┐    ┌──────┐    ┌──────┐    │
     ┌────┤  │ 思考  │───→│ 行动  │───→│ 观察  │   │
     │    │  └──────┘    └──────┘    └──┬───┘    │
     │    │  ┌──────────────────────────┘         │
     │    │  ↓                                    │
     │    │  ┌──────────────┐                     │
     │    │  │ 需要人类反馈？ │                     │
     │    │  └───────┬──────┘                     │
     │    │     否↓    ↓是                         │
     │    │  继续   ┌──────────┐                   │
     │    │  循环   │ 等待人类  │                   │
     │    │         └────┬─────┘                   │
     └────┘              ↓                         │
                    注入到下一轮循环                 │
          └──────────────────────────────────────┘
```

**问题 3: 如何实现渐进式降级？**

```
正常模式 (Full Autonomy):
  Agent 自主完成所有步骤

轻度降级 (Reduced Autonomy):
  每 3 步请求一次人工确认

中度降级 (Guided Mode):
  每步都请求人工确认，Agent 只提供建议

重度降级 (Manual Mode):
  Agent 仅提供信息，所有决策由人工做出
```

### 4.4 Harness 的设计模式

#### 模式 1: System Prompt 分层设计

```
┌─────────────────────────────────────────┐
│ 第1层: 基础身份 (Base Identity)          │  ← 不变
│ "你是一个代码助手，擅长Python和JS"        │
├─────────────────────────────────────────┤
│ 第2层: 行为规范 (Behavior Rules)         │  ← 很少变
│ "优先使用已有代码风格，不要过度注释"       │
├─────────────────────────────────────────┤
│ 第3层: 工具指令 (Tool Instructions)      │  ← 随工具变化
│ "你可以使用以下工具: Read, Write, Edit"   │
├─────────────────────────────────────────┤
│ 第4层: 动态上下文 (Dynamic Context)      │  ← 每次请求变化
│ "当前文件: main.py, 用户刚执行了测试"     │
├─────────────────────────────────────────┤
│ 第5层: 用户输入 (User Input)             │  ← 每次请求
│ "请帮我修复这个 bug"                      │
└─────────────────────────────────────────┘
```

**关键原则**: 
- 稳定的层放在前面（可以利用 Prompt Cache）
- 变化的层放在后面
- 每一层职责清晰，不互相干扰

#### 模式 2: 权限分级模型

```
┌───────────────────────────────────────────────────────┐
│                    权限分级                             │
│                                                        │
│  Level 0: 只读                                         │
│  ├── Read file, Search, List directory                 │
│  └── 自动执行，无需确认                                │
│                                                        │
│  Level 1: 写入（低风险）                                │
│  ├── Write file, Edit file                             │
│  └── 自动执行或简单确认                                │
│                                                        │
│  Level 2: 执行（中风险）                                │
│  ├── Run tests, Install packages                       │
│  └── 需要用户确认                                      │
│                                                        │
│  Level 3: 系统操作（高风险）                            │
│  ├── Delete file, Modify config, Network access        │
│  └── 需要详细确认 + 审计日志                           │
│                                                        │
│  Level 4: 禁止                                         │
│  ├── 访问敏感数据、执行危险命令                         │
│  └── 无论如何都不允许                                  │
└───────────────────────────────────────────────────────┘
```

#### 模式 3: 上下文窗口管理策略

```
上下文窗口 (例如 200K tokens)
┌──────────────────────────────────────────────────┐
│ System Prompt (固定)          ~2K tokens         │
├──────────────────────────────────────────────────┤
│ 工具定义 (固定)               ~3K tokens         │
├──────────────────────────────────────────────────┤
│ 历史对话 (滑动窗口)           ~50K tokens        │
│  ├── 最近 10 轮: 保留原文                       │
│  ├── 11-50 轮: 保留摘要                         │
│  └── 50 轮之前: 丢弃或进一步压缩                │
├──────────────────────────────────────────────────┤
│ 长期记忆 (按需加载)           ~10K tokens        │
│  ├── 检索相关记忆                               │
│  └── 注入到上下文                               │
├──────────────────────────────────────────────────┤
│ 工具结果 (最近)               ~30K tokens        │
├──────────────────────────────────────────────────┤
│ 当前任务上下文                ~50K tokens        │
├──────────────────────────────────────────────────┤
│ 预留空间 (输出)               ~55K tokens        │
└──────────────────────────────────────────────────┘
```

**三种压缩策略**:

| 策略 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **摘要压缩** | 用 LLM 将旧对话压缩为摘要 | 保留关键信息 | 可能丢失细节 |
| **滑动窗口** | 只保留最近 N 轮对话 | 简单高效 | 丢失早期上下文 |
| **选择性保留** | 根据相关性选择保留哪些内容 | 精准控制 | 实现复杂 |

### 4.5 Harness 与策略层的关系

Harness 是策略层的 **运行时载体**。同一个策略在不同 Harness 中的实现可能完全不同：

```
ReAct 策略的 Harness 实现对比:

┌──────────────┬──────────────────────────────────────────────┐
│ Claude Code  │ - System Prompt 注入代码操作规范               │
│   Harness    │ - 工具: Read/Write/Edit/Bash/Grep/Glob       │
│              │ - 权限: 分级确认 (写入/执行/危险操作)           │
│              │ - 上下文: 200K 窗口 + Prompt Cache             │
│              │ - 输出: 支持流式 + tool_use 结构化输出          │
├──────────────┼──────────────────────────────────────────────┤
│ ChatGPT      │ - System Prompt 注入通用助手人设               │
│   Harness    │ - 工具: Code Interpreter/DALL-E/Web Browse    │
│              │ - 权限: 沙箱执行，有限文件访问                  │
│              │ - 上下文: 128K 窗口                           │
│              │ - 输出: 流式文本 + 偶尔工具调用                 │
├──────────────┼──────────────────────────────────────────────┤
│ 自定义 Agent │ - System Prompt 完全自定义                    │
│   Harness    │ - 工具: 自定义注册                            │
│              │ - 权限: 自定义策略                             │
│              │ - 上下文: 取决于模型                           │
│              │ - 输出: 自定义解析                             │
└──────────────┴──────────────────────────────────────────────┘
```

### 4.6 Harness 工程的关键设计决策

| 决策点 | 选项 | 权衡 |
|--------|------|------|
| **工具注入方式** | 全量注入 vs 按需加载 | 全量简单但占上下文；按需省空间但增加延迟 |
| **权限确认** | 自动执行 vs 人工确认 | 自动快但有风险；确认安全但慢 |
| **上下文管理** | 滑动窗口 vs 摘要 vs RAG | 简单 vs 信息保留 vs 精准检索 |
| **错误处理** | 自动重试 vs 报错终止 | 自动恢复 vs 快速失败 |
| **输出格式** | 自由文本 vs 结构化 JSON | 灵活 vs 可靠解析 |
| **状态持久化** | 内存 vs 数据库 | 快 vs 可恢复 |

## 第五部分：三维协作与综合对比

### 5.1 三维如何协作？— 以 "代码生成 Agent" 为例

```
用户请求: "帮我实现一个 REST API，包含用户注册和登录功能"

═══════════════════════════════════════════════════════════════
Workflow (结构编排):
═══════════════════════════════════════════════════════════════
  DAG 定义:
    Step1: 分析需求 → Step2: 设计API  → Step3: 实现代码
                                          ↓
                                    Step4: 编写测试 → Step5: 验证运行

═══════════════════════════════════════════════════════════════
Harness (运行时 + 循环):
═══════════════════════════════════════════════════════════════
  静态环境:
    System Prompt: "你是一个Python后端开发专家..."
    工具: Read, Write, Edit, Bash(受限), Grep
    权限: 写入文件需确认，执行测试需确认
    上下文: 项目结构 + 现有代码 + 测试框架信息

  循环引擎 (Loop):
    最大步数: 20
    目标检查: 代码是否语法正确、测试是否通过
    反思机制: 测试失败时生成反思，改进代码
    预算控制: 总 Token 不超过 100K

═══════════════════════════════════════════════════════════════
策略 (推理决策):
═══════════════════════════════════════════════════════════════
  每个 Step 内部使用 ReAct:
    Thought: 需要先创建 Flask 应用的主文件
    Action: Write("app.py", flask_app_code)
    Observation: 文件创建成功
    Thought: 接下来创建用户模型...
    ...
```

### 5.2 三维的正交性

三维是 **正交的** — 每个维度的选择独立于其他维度：

```
┌───────────────────────────────────────────────────────────────┐
│  正交组合示例                                                   │
│                                                                │
│  组合1: DAG + 严格Harness(Goal-Loop) + ReAct                  │
│    → 结构化的代码生成系统                                       │
│                                                                │
│  组合2: 状态机 + 宽松Harness(Budget-Loop) + Reflexion          │
│    → 灵活的研究助手                                             │
│                                                                │
│  组合3: 事件驱动 + 企业Harness(Confidence-Loop) + Multi-Agent  │
│    → 企业级自动化平台                                           │
│                                                                │
│  组合4: Pipeline + 安全Harness(Fixed-Step) + Plan-Exec         │
│    → 数据处理流水线                                             │
└───────────────────────────────────────────────────────────────┘
```

### 5.3 三维常见误区

| 误区 | 正确理解 |
|------|---------|
| "选好策略就够了" | 策略只是三维之一，Workflow 和 Harness 设计同样重要 |
| "Harness 就是 SDK" | Harness 包含权限、上下文、循环引擎、可观测性等远超 SDK 的能力 |
| "Loop 是独立的一层" | Loop 是 Harness 的心脏，不是独立于 Harness 的层级 |
| "Workflow 和 Harness 是上下层" | 它们是正交的编织关系：Workflow 定义结构，Harness 填充运行时 |
| "Workflow 就是 DAG" | 状态机、事件驱动等模式在不同场景下各有优势 |
| "必须自己从零搭建" | 大部分场景可以基于成熟框架组合实现 |

## 第六部分：工程实践与选型指南

### 6.1 按场景的三维选型推荐

| 场景 | Workflow | Harness (含 Loop) | 策略 |
|------|----------|---------|------|
| **代码助手** | 状态机 | 代码操作权限 + 文件上下文 + Goal-Reflect Loop | ReAct |
| **客服机器人** | 状态机 | 知识库工具 + 对话历史 + Goal Loop | ReAct + RAG |
| **数据处理** | Pipeline/DAG | 数据工具 + 沙箱执行 + Fixed-Step Loop | Plan-Exec |
| **研究调研** | DAG | 搜索工具 + 文档管理 + Budget Loop | Plan-Exec + Reflexion |
| **内容创作** | 事件驱动 | 创作工具 + 风格指南 + Confidence Loop | ToT + CoT |
| **企业自动化** | 事件驱动 | 企业API + 权限管控 + Budget-Human Loop | Multi-Agent |
| **代码生成调试** | DAG | 代码工具 + 测试执行 + Reflect-Improve Loop | Reflexion + ReAct |

### 6.2 成本-效果-复杂度权衡

```
            效果
             ↑
             │       ╭── Multi-Agent + 事件驱动 + Budget
             │      ╭╯
             │     ╭╯   ╭── Plan-Exec + 状态机 + Reflect
             │    ╭╯   ╭╯
             │   ╭╯   ╭╯   ╭── ReAct + DAG + Goal
             │  ╭╯   ╭╯   ╭╯
             │ ╭╯   ╭╯   ╭╯   ╭── CoT + Pipeline + Fixed
             │╭╯   ╭╯   ╭╯   ╭╯
             ┼──────────────────────────→ 成本/复杂度
```

### 6.3 渐进式架构演进路线

```
Phase 1: MVP (最小可行产品)
├── Workflow: 简单 Pipeline 或无编排
├── Harness: 基础 Prompt + 少量工具 + Fixed-Step Loop
├── 策略: ReAct (默认选择)
└── 目标: 验证核心能力

Phase 2: 增强
├── Workflow: 添加条件分支和并行
├── Harness: 完善权限 + 上下文管理 + Goal-Based Loop + 停滞检测
├── 策略: 根据场景选择最优策略
└── 目标: 提升可靠性和质量

Phase 3: 生产化
├── Workflow: 状态机或事件驱动
├── Harness: 完整可观测性 + 审计 + Budget-Confidence-Human Loop
├── 策略: Multi-Agent (如需要)
└── 目标: 生产级稳定性和可扩展性
```

### 6.4 一句话总结

> **Workflow 编排多个 Agent，Harness 定义单个 Agent 怎么跑（含循环），策略决定每一步怎么想。**
>
> Workflow 和 Harness 不是上下层，是同一系统的两个正交视角 —— Workflow 看结构，Harness 看运行。Loop 是 Harness 的心脏，不是独立层。
